# ugpibd v0.2.0 — safer defaults and 82357A firmware

Status: approved design (pending spec review)
Date: 2026-07-20

## Summary

Two groups of changes, released together as `0.2.0`:

1. **Defaults / CLI cleanup** — bind to localhost, unify the default-address
   arg, make Prologix opt-in, quiet the default log format.
2. **Agilent 82357A firmware loading** — bundle the A's firmware image and
   parameterize the EZ-USB reset register so a cold 82357A can be brought up
   in-process, just like the 82357B.

Nothing here changes the on-bus protocol or the front-end wire protocols; the
work is startup configuration and a small generalization of the firmware
uploader.

---

## Part 1 — Defaults / CLI cleanup

### 1.1 Bind to localhost by default

`--bind` default changes `0.0.0.0` → `127.0.0.1`. Both listeners (HiSLIP and,
when enabled, Prologix) inherit it as today. Users who want remote access pass
`--bind 0.0.0.0` (or a specific interface address) explicitly.

### 1.2 Unify `--default-address`

Rename `--hislip-default-pad` → `--default-address`, **default `0`**, and use it
for both front-ends:

- HiSLIP: the fallback primary address when a client's subaddress does not
  encode one (was the sole consumer; its default moves 14 → 0).
- Prologix: seeds `PrologixState`'s initial `addr` (already 0, unchanged).

Consequence to document: a HiSLIP client using the bare `hislip0` subaddress now
targets pad 0 unless `--default-address` is set.

`src/main.rs` is the only code consumer; `README.md` line ~72 is the only doc
reference.

### 1.3 Prologix opt-in

New flag `--enable-prologix` (bool, default `false`). HiSLIP stays enabled by
default (its `--hislip-port` default is unchanged). Implementation mirrors the
existing optional-HiSLIP pattern in `main.rs`:

- Build the Prologix `TcpListener` only when `--enable-prologix` is set;
  otherwise its branch of the `tokio::select!` is `std::future::pending()`.
- If **neither** front-end is enabled (`--hislip-port 0` and no
  `--enable-prologix`), exit with a clear error rather than idling silently.

### 1.4 Quieter default log format

Default level stays `ugpibd=info` (still reports backend/adapter discovery,
listeners bound, client connect/disconnect, and warnings/errors). The
`tracing_subscriber` formatter changes to reduce per-line noise:

- `.with_target(false)` — drop the `ugpibd::backend::agilent_82357::usb:` prefix.
- A compact time format (`.without_time()` or a short `%H:%M:%S` timer) so lines
  read like `INFO using backend agilent-82357b` instead of a full RFC3339 stamp
  plus module path.

`-v`/`-vv` and the `RUST_LOG` override are unchanged. This is formatter-only; no
log call sites move between levels.

### 1.5 Version

`Cargo.toml` `version = "0.1.0"` → `"0.2.0"`; update `Cargo.lock`. Minor bump:
additive feature plus a couple of intentional default/CLI changes, pre-1.0.

## Part 2 — Agilent 82357A firmware loading

### 2.1 Background

The 82357A is a first-generation Cypress EZ-USB (AN2131 / "FX") part. Its 8051
reset register (CPUCS) is at **`0x7F92`**, versus the 82357B's FX2 **`0xE600`**.
Confirmed against the upstream loader (`fmhess/linux_gpib_firmware`,
`agilent_82357a/firmware.c`) and the upstream README: the A is loaded by fxload
**without** `-t fx2` and **without** a second-stage loader, i.e. the exact same
`ANCHOR_LOAD_INTERNAL` (vendor request `0xA0`) sequence we already use for the B
— only the CPUCS reset address differs. So no external-RAM / two-stage path is
needed.

### 2.2 Changes

1. **Bundle `firmware/82357a_fw.hex`** from the same upstream project and the
   same redistribution terms as the existing B image. Extend `firmware/LICENSE`
   to cover the new file (it is for the 82357A; `measat_releaseX1.8.hex` is for
   the 82357B).
2. **Parameterize the reset register.** Add `cpucs_addr: u16` to
   `agilent_82357::Model`: `0xE600` for `MODEL_82357B`, `0x7F92` for
   `MODEL_82357A`. Thread it into `firmware::upload_firmware(...)` (replacing the
   hardcoded `CPUCS_ADDR`). The `0xA0` request and record-writing loop are
   unchanged. Rename the FX2-specific symbols to neutral names where they now
   cover both parts (e.g. keep `ANCHOR_LOAD_INTERNAL`; the `fx2_control` helper
   and `FX2_FIRMWARE_82357B` naming get generalized/renamed).
3. **Set `MODEL_82357A.firmware = Some(...)`** pointing at the bundled A image.
4. The existing double-upload retry loop in `initialize_device` is a B-specific
   quirk; for the A it harmlessly succeeds on the first attempt (renumeration to
   `0x0107`), so no branching is needed.
5. Update `README.md` (the adapter table and the 82357A note) and the module doc
   comments in `agilent_82357/mod.rs` / `firmware.rs`: the A is no longer
   "firmware not bundled." It remains **hardware-untested**; keep that caveat.

## Testing

- **Firmware:**
  - Parse the bundled `82357a_fw.hex` via `parse_hex` — validates the blob loads
    and is non-empty.
  - Guard `MODEL_82357A.firmware.is_some()` and the per-model CPUCS addresses
    (`A == 0x7F92`, `B == 0xE600`).
- **Defaults:** no automated coverage beyond a smoke check that `--help` reflects
  the renamed/added flags; the CLI wiring is exercised by the existing test
  suites continuing to build and pass.
- **Existing suites** (`parse_hex`, prologix, hislip client/integration, scpi)
  must still pass. Full run: `cargo test`, plus `cargo clippy` and `cargo fmt`.

## Out of scope

- Actual on-hardware validation of the 82357A path (no hardware available).
